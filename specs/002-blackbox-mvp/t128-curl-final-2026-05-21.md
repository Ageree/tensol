### Step 5 schema mismatch (minor, non-blocking)

The runner sent `{ "entries": [...] }`; the API expects `{ "attack_surface": [...] }`:

```json
{
  "error": "validation_error",
  "details": [
    { "field": "attack_surface", "message": "Required" },
    { "field": "body", "message": "Unrecognized key(s) in object: 'entries'" }
  ]
}
```

The draft remained launchable because `primary_domain` alone satisfied the
launch precondition. Fixing the runner key is trivial — but the test still
proved the underlying launch path.

### Diagnostic probes (from inside the container, same SA key the server uses)

|------------------------------------------|--------|

| `POST /compute/v1/instances` **with** `oneToOneNatSpec` | **403** — same error as production |
| `POST /compute/v1/instances` **without** public IP, with WRONG image | 400 image-not-found (passes IAM check) |
| `POST /compute/v1/instances` **without** public IP, with **correct** image | **200** — VM `fhmqq9tnts2aqub9a4ee` created and torn down |

This isolates the failure to **one specific IAM permission**: SA
`ajebdg35q6f107q3jl9k` has `compute.editor + vpc.user + vpc.externalAddresses.user
+ resource-manager.viewer + storage.editor + storage.uploader +
iam.serviceAccounts.user` bound at folder level, but **NAT / public-IPv4
allocation via `oneToOneNatSpec` is denied**.

## New blockers found

1. **OPERATOR**: SA `ajebdg35q6f107q3jl9k` needs an IAM role that
   authorises public-IP/NAT allocation during instance create.
   `vpc.externalAddresses.user` (already bound) only allows ATTACHING an
   already-reserved static IP, not creating ephemeral NAT addresses. Likely
   missing: `vpc.publicAdmin` or cloud-level `compute.editor`. Alternative:
   pre-reserve a pool of static external addresses and pass `address` in the
   NAT spec instead of letting Yandex allocate.
2. **CODE**: stale Ubuntu image ID in `server/src/vps/yandex.ts:325`.
   Refresh to `fd85f16aij1rvpce9gas` (or wire dynamic family lookup).
3. **TEST FIXTURE**: curl runner step 5 sent wrong key shape (`entries` vs
   `attack_surface`). Cosmetic; not a server bug.

## Final verdict

End-to-end happy-path **NOT** achieved — but for a different reason than
last attempt:
- Previously (2026-05-19, T047), the failure was at IAM-token issuance level
  (SA had no folder roles).
- Today the IAM/token path is fully repaired, and we proved that VM create
  **does** work for the SA when the network spec omits ephemeral NAT.
- The remaining blocker is one missing IAM bit specific to public-IP
  allocation. Cannot be fixed from curl/runtime — needs `yc resource-manager
  folder add-access-binding` (or cloud-level) by operator.

Every API surface in front of `spawnVm` is green: auth issue/poll, telegram
webhook, scan-order CRUD, dns-verify, launch, scan-status polling, audit
chain, quota refund. The product loop is bytes-from-flushing — held only by
one IAM grant + one stale image ID.
