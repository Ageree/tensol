# Phase 1 Data Model — Backend v2

SQLite + Drizzle. Single database file `server/data/tensol.db` in production, `:memory:` in tests.

All timestamps are integers (Unix epoch milliseconds) per SQLite idiom; Drizzle exposes them as `Date` objects when annotated `{ mode: 'timestamp_ms' }`.

All primary keys are 26-character ULIDs (lexicographically sortable, URL-safe). Generated by `lib/ids.ts`.

## Entities

### `users`

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PK | ULID |
| `email` | TEXT UNIQUE NOT NULL | Lowercased on insert |
| `created_at` | INTEGER NOT NULL | Epoch ms |

**Indexes**: `email` (unique).

**Lifecycle**: Created on first successful magic-link verification for an unseen email. Never deleted from the API.

---

### `sessions`

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PK | 256-bit random hex (server-side session token) |
| `user_id` | TEXT NOT NULL FK → users.id | |
| `created_at` | INTEGER NOT NULL | |
| `expires_at` | INTEGER NOT NULL | created_at + 30 days |

**Indexes**: `user_id`, `expires_at`.

**Lifecycle**: Created on magic-link verification. Deleted on logout or expiry (sweeper job).

---

### `magic_link_tokens`

| Column | Type | Notes |
|--------|------|-------|
| `token` | TEXT PK | 256-bit random hex |
| `email` | TEXT NOT NULL | Target inbox |
| `expires_at` | INTEGER NOT NULL | created_at + 15 min |
| `used_at` | INTEGER NULL | Set on successful redemption |

**Indexes**: `email`, `expires_at`.

**Lifecycle**: Created on `POST /api/auth/request-link`. Burnt on first verification. Swept hourly.

---

### `projects`

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PK | ULID |
| `user_id` | TEXT NOT NULL FK → users.id | Owner |
| `name` | TEXT NOT NULL | 1–80 chars |
| `created_at` | INTEGER NOT NULL | |

**Indexes**: `user_id`.

---

### `targets`

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PK | ULID |
| `project_id` | TEXT NOT NULL FK → projects.id | |
| `url` | TEXT NOT NULL | Normalized: lowercased host, no trailing slash |
| `status` | TEXT NOT NULL | `unverified` \| `verified` \| `expired` |
| `verified_at` | INTEGER NULL | When `status` last became `verified` |
| `created_at` | INTEGER NOT NULL | |

**Indexes**: `project_id`, `status`.

**State transitions**:

| From | Event | To |
|------|-------|----|
| unverified | auth-proof succeeds | verified |
| verified | 90 days since `verified_at` (computed) | expired |
| expired | auth-proof succeeds | verified |

State `expired` is **computed** from `verified_at` at read time — the column stays at `verified` until a new verification or until an explicit "rotate stale" job marks it. To keep things simple, we only test `(status='verified' AND now - verified_at < 90 days)` at scan-start; no background sweeper for this.

### `auth_proofs`

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PK | ULID |
| `target_id` | TEXT NOT NULL FK → targets.id | |
| `challenge` | TEXT NOT NULL | `tensol-verify=<32-byte-hex>` |
| `method` | TEXT NULL | `dns_txt` \| `file` \| `meta_tag`; set on success |
| `status` | TEXT NOT NULL | `pending` \| `verified` \| `expired` |
| `created_at` | INTEGER NOT NULL | |
| `verified_at` | INTEGER NULL | |
| `expires_at` | INTEGER NOT NULL | created_at + 24 h |

**Indexes**: `target_id`, `status`, `expires_at`.

**Lifecycle**: One row per challenge issuance. Multiple rows may exist for the same target if user retries; only the most recent matters.

### `scans`

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PK | ULID |
| `user_id` | TEXT NOT NULL FK → users.id | Denormalized for cheap queries |
| `target_id` | TEXT NOT NULL FK → targets.id | |
| `profile` | TEXT NOT NULL | `recon` \| `standard` \| `max` |
| `status` | TEXT NOT NULL | `queued` \| `running` \| `completed` \| `failed` \| `cancelled` |
| `failure_reason` | TEXT NULL | Set when `status='failed'` |
| `started_at` | INTEGER NOT NULL | Set when status moves to `queued` |
| `completed_at` | INTEGER NULL | Set on any terminal state |
| `usage_tokens` | INTEGER NULL | Reported by VPS agent |
| `usage_usd_cents` | INTEGER NULL | Reported by VPS agent |

**Indexes**: `user_id`, `target_id`, `status`.

**State transitions**:

```
queued → running → completed
                 → failed
                 → cancelled

queued → cancelled  (cancelled before VPS provisioned)
queued → failed     (provisioner gave up)
```

### `findings`

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PK | ULID |
| `scan_id` | TEXT NOT NULL FK → scans.id | |
| `severity` | TEXT NOT NULL | `critical` \| `high` \| `medium` \| `low` \| `info` |
| `title` | TEXT NOT NULL | 1–200 chars |
| `body_md` | TEXT NOT NULL | Markdown |
| `evidence_json` | TEXT NULL | JSON blob; agent-supplied |
| `created_at` | INTEGER NOT NULL | |
| `dedup_key` | TEXT NOT NULL | `scan_id + ':' + sha256(title)` |

**Indexes**: `scan_id`, `dedup_key` (unique).

**Idempotency**: Insert uses `INSERT INTO findings ... ON CONFLICT(dedup_key) DO NOTHING`. Duplicate callback bodies don't double-store.

### `audit_log`

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER PK AUTOINCREMENT | Monotonic |
| `ts` | INTEGER NOT NULL | Epoch ms — appears in canonical message as ISO 8601 |
| `event` | TEXT NOT NULL | e.g. `scan_started`, `decepticon_invoked` |
| `user_id` | TEXT NULL | |
| `project_id` | TEXT NULL | |
| `target_id` | TEXT NULL | |
| `scan_id` | TEXT NULL | |
| `vps_instance_id` | TEXT NULL | |
| `auth_proof_id` | TEXT NULL | |
| `finding_id` | TEXT NULL | |
| `severity` | TEXT NULL | |
| `outcome` | TEXT NOT NULL | `success` \| `failure` \| `rejected` |
| `metadata_json` | TEXT NOT NULL | Alpha-sorted keys |
| `prev_signature` | TEXT NOT NULL | hex of previous row's signature (empty string for row #1) |
| `signature` | TEXT NOT NULL | hex HMAC-SHA256 over 13-field canonical message |

**Indexes**: `scan_id`, `event`, `ts`.

**Invariants**:

- Inserts are serialized through `emitSignedAudit()` which begins an immediate transaction and reads `MAX(id)` for `prev_signature`.
- No updates or deletes allowed (enforced at application layer; SQLite has no row-level DDL trigger we rely on).

### `vps_instances`

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PK | ULID |
| `scan_id` | TEXT NOT NULL FK → scans.id | |
| `provider` | TEXT NOT NULL | `hetzner` (only one supported in v2) |
| `provider_server_id` | TEXT NOT NULL | Hetzner's numeric ID, stringified |
| `ipv4` | TEXT NULL | Set when provisioning completes |
| `status` | TEXT NOT NULL | `provisioning` \| `alive` \| `tearing_down` \| `destroyed` |
| `sign_key` | TEXT NOT NULL | base64 32 bytes — used to HMAC-sign callback |
| `created_at` | INTEGER NOT NULL | |
| `destroyed_at` | INTEGER NULL | |

**Indexes**: `scan_id` (unique), `status`.

**Invariants**:

- Exactly one `vps_instances` row per `scans` row that ever reached `running` status.
- `sign_key` is generated by the backend at insert time and embedded into the cloud-init payload sent to Hetzner — the key never leaves the database after that.

### `jobs`

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PK | ULID |
| `type` | TEXT NOT NULL | `spawn_vps` \| `dispatch_scan` \| `watchdog_scan` \| `teardown_vps` |
| `payload_json` | TEXT NOT NULL | Type-specific JSON |
| `status` | TEXT NOT NULL | `pending` \| `running` \| `done` \| `failed` |
| `scheduled_at` | INTEGER NOT NULL | Earliest eligibility (epoch ms) |
| `attempts` | INTEGER NOT NULL DEFAULT 0 | |
| `last_error` | TEXT NULL | |
| `created_at` | INTEGER NOT NULL | |
| `updated_at` | INTEGER NOT NULL | |

**Indexes**: `status, scheduled_at` (composite, the runner's hot path), `type`.

**Worker contract**:

- `BEGIN IMMEDIATE; SELECT … WHERE status='pending' AND scheduled_at <= now() ORDER BY scheduled_at LIMIT 1; UPDATE jobs SET status='running' WHERE id=…; COMMIT;`
- Handler runs outside the transaction.
- On success: `UPDATE jobs SET status='done', updated_at=now()`.
- On exception: `UPDATE jobs SET status='pending', attempts=attempts+1, scheduled_at=now()+pow(2,attempts)*1000, last_error=msg`. After 5 attempts → status `failed`.

## ERD (textual)

```
users ─< sessions
users ─< magic_link_tokens (logically by email; no FK because email may not yet be a user)
users ─< projects ─< targets ─< auth_proofs
users ─< scans >─ targets
scans ─< findings
scans ─ 1:1 ─ vps_instances
audit_log references any of: users, projects, targets, scans, vps_instances, auth_proofs, findings (nullable FKs)
jobs ── (free-standing) ── references scan IDs by payload only, no FK to keep operational flexibility
```

## Migration shape

`migrations/0000_init.sql` creates all nine tables above in a single transaction. Drizzle generates it from `src/db/schema.ts`. No data migration from v1 (clean slate).
